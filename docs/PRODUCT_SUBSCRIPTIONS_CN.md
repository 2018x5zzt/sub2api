# 产品订阅模块设计与排查记录

**日期：** 2026-05-02

## 目标

产品订阅模块用于恢复并扩展 xlabapi 的“新版产品”订阅能力。它和 Upstream 的旧版订阅不是同一种模型：旧版订阅是“一个兑换码激活一个分组限额”，新版产品是“一个产品激活多个真实分组，这些分组共享同一份产品额度，并支持昨日额度结转”。

本轮实现还新增一个默认关闭的余额兜底能力：当用户订阅额度耗尽时，用户可以选择自动消耗余额，并设置余额兜底消耗上限。同时补齐余额模式下的 `team/plus` 混合号池分组，倍率为 1 倍，为订阅 `team/plus` 混合号池提供明确的余额兜底目标。

## 核心概念

### 旧版订阅

旧版订阅围绕 `user_subscriptions` 和 `groups.subscription_type = 'subscription'` 工作。一个兑换码或订阅计划通常只绑定一个分组，API Key 也只在该分组限额内计费。旧版订阅的日、周、月限额直接来自分组字段。

旧版订阅的问题是它无法表达“一个产品解锁多个分组，但共享同一个额度池”。如果继续把产品强行投影成旧版订阅，会导致多个分组各自拥有独立额度，和产品设计冲突。

### 新版产品

新版产品由三张核心表组成：

- `subscription_products`：产品定义，包括产品编码、名称、有效天数、日/周/月额度和排序。
- `subscription_product_groups`：产品可见分组绑定，包括真实 `group_id` 和该分组消耗产品额度时使用的 `debit_multiplier`。
- `user_product_subscriptions`：用户已激活的产品订阅实例，保存共享用量、过期时间和昨日结转额度。

API Key 仍然绑定真实 `groups.id`，不会绑定产品 ID。用户购买或兑换一个产品后，前端创建 API Key 时应能看到该产品暴露的多个真实分组。请求进入网关后，系统根据用户、API Key 分组和产品绑定解析产品订阅上下文，再把本次费用按分组倍率折算后扣到同一个产品订阅额度池。

## 多分组共享额度

产品可以同时绑定多个分组。例如一个“每日 45 美元”的周卡可以绑定：

- `team/plus` 混合号池，产品消耗倍率 1 倍。
- `image` 生图号池，产品消耗倍率按产品配置。
- `pro` 号池，产品消耗倍率按产品配置。

用户为不同分组生成不同 API Key，请求会走对应真实分组的账号池和调度规则，但产品用量写入同一个 `user_product_subscriptions` 记录。产品额度展示也必须按产品维度展示，不能把每个分组展示成独立订阅额度。

用户激活产品后，前端需要提示用户生成一个新的产品分组 API Key，避免用户继续使用余额模式 API Key 而误扣余额。

## 昨日结转

结转规则是：昨天未用完的产品日额度最多保留一天到今天，今天优先消耗昨日结转额度，昨天结转额度今天没用完不能继续保留到明天。

系统用以下字段表达结转：

- `daily_carryover_in_usd`：今天带入的昨日可结转总额。
- `daily_carryover_remaining_usd`：今天尚未消耗的昨日结转剩余额。
- `daily_usage_usd`：今天总产品消耗，包含结转部分和今天新额度部分。

前端展示每日进度条时应使用：

- 总可用额度：`daily_limit_usd + daily_carryover_in_usd`
- 已用额度：`daily_usage_usd`
- 今日新额度已用：`max(daily_usage_usd - (daily_carryover_in_usd - daily_carryover_remaining_usd), 0)`
- 昨日结转已用：`daily_carryover_in_usd - daily_carryover_remaining_usd`

页面小字应明确说明：进度条包含今日产品额度和昨日结转额度，并且昨日结转只保留一天。

## 产品选择与自动切换

当前代码中存在一个需要修复的风险：`GetActiveProductSubscriptionByUserAndGroupID` 对同一用户同一分组只按 `expires_at DESC, id DESC` 取一个产品订阅。用户如果同时有 45 美元周卡和 150 美元月卡，并且两个产品绑定相同真实分组，系统可能优先选到过期时间更晚的产品，而不是按预期优先消耗先激活或优先级更高的产品。

新版规则如下：

1. 系统应查询用户在该真实分组下所有活跃产品订阅，而不是只取一条。
2. 产品只能在同一个 `product_family` 内自动切换消耗。默认产品族为 `default`。
3. 同一产品族内按 `subscription_products.sort_order ASC`、`user_product_subscriptions.starts_at ASC`、`user_product_subscriptions.id ASC` 选择第一个还有额度的产品订阅。
4. 第一个产品订阅额度耗尽后，系统自动尝试同产品族内的下一个产品订阅。
5. 不同产品族之间不能自动切换，避免未来新增不同用途产品时发生越权消耗。
6. `ResolveActiveProductByGroupID` 只能作为旧兼容兜底。商业售卖和兑换码生成必须使用明确的 `product_id`，不能只用 `group_id` 猜产品。

产品族字段需要落库到 `subscription_products.product_family`。已有产品统一迁移为 `default`，后续新产品如果不能共享自动切换，应设置不同的产品族。

## 余额兜底

余额兜底是用户级开关，默认关闭。字段设计：

- `users.subscription_balance_fallback_enabled`：是否开启“订阅消耗完时，自动消耗余额”。
- `users.subscription_balance_fallback_limit_usd`：余额兜底累计上限，必须大于 0 才会生效。
- `users.subscription_balance_fallback_used_usd`：已经通过订阅兜底消耗的余额金额。

本轮把上限定义为累计上限，不做每日或每月自动重置。用户关闭开关时不清空已使用值；用户调高上限后，剩余额度等于 `limit - used`。管理员未来可以单独做重置或周期策略。

余额兜底只在以下情况生效：

1. API Key 绑定的是订阅分组。
2. 用户在该订阅分组下没有可继续消耗的同产品族产品额度，或者产品已过期/不存在。
3. 用户已开启余额兜底，且兜底上限大于 0，且已使用兜底金额小于上限。
4. 当前订阅分组存在明确的余额分组映射。
5. 用户余额大于 0。

兜底请求进入余额计费模式：本次 usage log 的 `billing_type` 为余额模式，不再写入产品订阅用量，不再消耗旧版订阅限额。实际扣费使用余额分组倍率和用户专属分组倍率，避免订阅分组倍率影响余额价格。

由于实际费用只有请求结束后才能准确计算，系统在请求前只能做“是否还有兜底额度”的资格检查。扣费落库时必须原子扣减余额并累加 `subscription_balance_fallback_used_usd`，同时校验不会超过上限。若一次请求的实际费用超过剩余额度，扣费事务应失败并返回计费错误；这比静默透支兜底上限更安全。

## 订阅分组到余额分组的映射

余额兜底不能直接继续使用订阅分组，因为订阅分组的账号池和倍率代表订阅产品，不代表余额计费产品。需要在分组层显式配置映射：

- 新增 `groups.balance_fallback_group_id`，仅对订阅分组有意义。
- 当订阅额度耗尽且用户开启余额兜底时，系统把本次请求的运行分组切换到 `balance_fallback_group_id` 指向的余额分组。
- 目标余额分组必须是活跃分组，且 `subscription_type = 'standard'`。

本轮新增余额模式 `team/plus` 混合号池，倍率为 1 倍。迁移只创建或修正分组与映射，不迁移现有 API Key 或账号池；账号池和 API Key 的批量迁移由后续任务处理。

`pro` 分组已经有明确的订阅/余额对应关系时，应通过 `balance_fallback_group_id` 显式配置；如果当前数据库无法通过稳定名称识别已有 `pro` 余额分组，则迁移不应猜测，管理员后续可以在分组后台配置映射。

## 前端体验

用户订阅页需要同时完成销售说明和运行状态说明：

- 展示产品订阅支持多分组、共享额度和昨日结转。
- 每个产品卡片展示总每日进度条，包含昨日结转。
- 进度条下方展示今日额度、昨日结转、昨日结转剩余和各分组倍率。
- 激活产品或兑换产品成功后提示用户去生成新的产品分组 API Key。
- 增加默认关闭的开关“订阅消耗完时，自动消耗余额”。
- 开关开启时展示“余额兜底上限”输入框，并显示已使用和剩余额度。

管理员侧需要能继续：

- 创建产品订阅码时绑定明确 `product_id`。
- 查看所有产品订阅用户。
- 查看用户产品订阅自激活以来的用量统计，包括昨日结转组成。

## 迁移兼容

- 旧 API Key 仍按真实 `group_id` 鉴权。绑定订阅分组的旧 Key 可以命中新版产品订阅上下文，但前端仍应引导用户新建产品分组 Key。
- 未使用的兑换码如果绑定 `product_id`，激活新版产品；如果只绑定旧版订阅字段，则继续按旧版订阅处理，不做隐式产品猜测。
- 已迁移到新版产品订阅的用户，前端优先展示新版产品订阅，旧版订阅只作为兼容信息保留。
- 不在本轮迁移账号池和 API Key，不批量更改用户现有 Key 的 `group_id`。

## 测试计划

后端测试：

- 用户同一分组下有两个同产品族活跃产品，先选择排序/激活更早的产品；该产品额度耗尽后自动选择下一个产品。
- 用户同一分组下有不同产品族产品，不跨产品族自动切换。
- 产品选择不再按最晚过期时间决定。
- 余额兜底默认关闭时，产品额度耗尽仍返回限额错误。
- 余额兜底开启且有剩余额度时，订阅分组请求切换到映射的余额分组并按余额计费。
- 余额兜底扣费事务同时扣余额和累加 `subscription_balance_fallback_used_usd`，超过上限时失败。
- 新迁移能创建余额 `team/plus` 混合号池，倍率为 1，且不修改账号池/API Key。

前端测试：

- 订阅页展示包含结转的每日进度条和组成说明。
- 订阅页展示余额兜底开关，默认关闭。
- 开启兜底后可以设置上限，保存后重新加载仍保持。
- 兑换产品成功后提示生成新的产品分组 API Key。
