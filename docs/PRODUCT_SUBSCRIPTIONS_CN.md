# 产品订阅模块设计与排查记录

**日期：** 2026-05-04

## 目标

产品订阅模块用于恢复并扩展 xlabapi 的“新版产品”订阅能力。它和 Upstream 的旧版订阅不是同一种模型：旧版订阅是“一个兑换码激活一个分组限额”，新版产品是“一个产品激活多个真实订阅分组，这些分组共享同一份产品额度，并支持昨日额度结转”。

最终行为必须是闭环且显式的产品订阅计费：

- 产品订阅兑换码必须绑定明确的 `product_id`。缺少 `product_id` 是配置错误，生成和兑换都必须拒绝，不能通过 `group_id` 猜产品。
- API Key 绑定真实 `groups.id`，并可额外绑定 `subscription_product_family`。当同一个订阅分组下有多个可见产品族时，必须显式选择产品族。
- 余额兜底由用户选择，默认关闭。用户必须开启、选择有权限的活跃标准分组，并设置正数累计上限后才会生效。
- 产品日额度窗口和昨日结转按北京时间 UTC+8 的 00:00 重置。
- 产品周/月额度窗口是从订阅实例窗口开始计算的滚动 7/30 天窗口，不是自然周或自然月。
- 产品绑定只能指向订阅分组，不能绑定标准余额分组。
- 管理员产品订阅列表需要支持调整、重置额度、撤销订阅，并支持管理用户的余额兜底设置。

## 核心概念

### 旧版订阅

旧版订阅围绕 `user_subscriptions` 和 `groups.subscription_type = 'subscription'` 工作。一个兑换码或订阅计划通常只绑定一个分组，API Key 也只在该分组限额内计费。旧版订阅的日、周、月限额直接来自分组字段。

旧版订阅的问题是它无法表达“一个产品解锁多个分组，但共享同一个额度池”。如果继续把产品强行投影成旧版订阅，会导致多个分组各自拥有独立额度，和产品设计冲突。

旧版订阅仍可服务旧分组和旧流程，但新版产品订阅解析失败时不能隐式回退到旧版 `user_subscriptions`。

### 新版产品

新版产品由三张核心表组成：

- `subscription_products`：产品定义，包括产品编码、名称、有效天数、日/周/月额度、排序和 `product_family`。
- `subscription_product_groups`：产品可见分组绑定，包括真实 `group_id` 和该分组消耗产品额度时使用的 `debit_multiplier`。
- `user_product_subscriptions`：用户已激活的产品订阅实例，保存共享用量、过期时间、额度窗口和昨日结转额度。

API Key 仍然绑定真实 `groups.id`，不会绑定产品 ID。用户购买或兑换一个产品后，前端创建 API Key 时应能看到该产品暴露的多个真实订阅分组。请求进入网关后，系统根据用户、API Key 分组和 API Key 的产品族选择解析产品订阅上下文，再把本次费用按分组倍率折算后扣到同一个产品订阅额度池。

## 多分组共享额度

产品可以同时绑定多个订阅分组。例如一个“每日 45 美元”的周卡可以绑定：

- `team/plus` 混合号池，产品消耗倍率 1 倍。
- `image` 生图号池，产品消耗倍率按产品配置。
- `pro` 号池，产品消耗倍率按产品配置。

用户为不同订阅分组生成不同 API Key，请求会走对应真实分组的账号池和调度规则，但产品用量写入同一个 `user_product_subscriptions` 记录。产品额度展示也必须按产品维度展示，不能把每个分组展示成独立订阅额度。

产品绑定只能指向活跃的订阅分组，也就是 `groups.subscription_type = 'subscription'` 的分组。后端必须拒绝把标准余额分组绑定到产品，前端分组选择器也应只展示可绑定的订阅分组。

用户激活产品后，前端需要提示用户生成新的产品订阅分组 API Key，避免用户继续使用余额模式 API Key 而误扣余额。

## 额度窗口与昨日结转

### 日额度窗口

产品日额度窗口按北京时间 UTC+8 计算，每天 00:00 重置。无论服务器部署时区是什么，产品日额度、每日重置和昨日结转都必须使用北京时间窗口。

结转规则是：昨天未用完的产品日额度最多保留一天到今天，今天优先消耗昨日结转额度，昨天结转额度今天没用完不能继续保留到明天。

系统用以下字段表达结转：

- `daily_carryover_in_usd`：今天带入的昨日可结转总额。
- `daily_carryover_remaining_usd`：今天尚未消耗的昨日结转剩余额。
- `daily_usage_usd`：今天总产品消耗，包含结转部分和今天新额度部分。

前端展示每日进度条时应使用：

- 总可用额度：`daily_limit_usd + daily_carryover_in_usd`
- 已用额度：`daily_usage_usd`
- 今日新额度已用：`max(daily_usage_usd - (daily_carryover_in_usd - daily_carryover_remaining_usd), 0)`
- 昨日结转已用：`daily_carryover_in_usd - daily_carryover_remaining_usd`

页面小字应明确说明：每日窗口按北京时间 00:00 重置，进度条包含今日产品额度和昨日结转额度，并且昨日结转只保留一天。

### 周/月额度窗口

产品周额度和月额度不是自然日历窗口：

- 周额度窗口是从订阅实例的周窗口开始时间起算的滚动 7 天窗口。
- 月额度窗口是从订阅实例的月窗口开始时间起算的滚动 30 天窗口。

例如订阅在北京时间 5 月 4 日 15:30 激活，则周窗口从该实例的窗口起点滚动 7 天，月窗口滚动 30 天；不能按周一到周日或每月 1 日到月底重置。

管理员重置额度时也必须遵守同一语义：重置日额度时重置北京时间日窗口和结转；重置周/月额度时重置对应实例的滚动 7/30 天窗口。

## 产品选择与自动切换

当前代码中曾存在一个需要修复的风险：`GetActiveProductSubscriptionByUserAndGroupID` 对同一用户同一分组只按 `expires_at DESC, id DESC` 取一个产品订阅。用户如果同时有 45 美元周卡和 150 美元月卡，并且两个产品绑定相同真实分组，系统可能优先选到过期时间更晚的产品，而不是按预期优先消耗先激活或优先级更高的产品。

新版规则如下：

1. 系统应查询用户在该真实订阅分组下所有活跃产品订阅，而不是只取一条。
2. 产品只能在同一个 `product_family` 内自动切换消耗。默认产品族为 `default`。
3. API Key 可绑定 `api_keys.subscription_product_family`。设置后，只能在该产品族内解析产品订阅。
4. 如果 API Key 没有绑定产品族，并且该用户在该订阅分组下只有一个可用产品族，系统可以使用该产品族。
5. 如果 API Key 没有绑定产品族，并且该用户在该订阅分组下有多个可用产品族，请求必须失败并返回明确的产品族歧义错误，前端创建或编辑 API Key 时也必须要求用户选择产品族。
6. 同一产品族内按 `subscription_products.sort_order ASC`、`user_product_subscriptions.starts_at ASC`、`user_product_subscriptions.id ASC` 选择第一个还有额度的产品订阅。
7. 第一个产品订阅额度耗尽后，系统自动尝试同产品族内的下一个产品订阅。
8. 不同产品族之间不能自动切换，避免未来新增不同用途产品时发生越权消耗。
9. 产品订阅分组解析失败时不能回退到旧版订阅。
10. `ResolveActiveProductByGroupID` 只能作为旧兼容辅助查询。商业售卖、兑换码生成和兑换必须使用明确的 `product_id`，不能只用 `group_id` 猜产品。

产品族字段落库在 `subscription_products.product_family`。已有产品统一迁移为 `default`，后续新产品如果不能共享自动切换，应设置不同的产品族。

## 产品订阅兑换码

产品订阅兑换码必须闭环到产品定义：

1. 管理员生成产品订阅兑换码时必须选择产品，后端必须要求 `product_id`。
2. 前端在未选择产品时应禁用或阻止产品订阅码生成。
3. 兑换产品订阅码时，如果兑换码缺少 `product_id`，即使带有历史 `group_id`，也必须拒绝并返回配置错误。
4. 系统不能从 `group_id`、产品绑定或分组名称猜测产品。
5. 余额码、并发码和旧版订阅码不受产品 `product_id` 强制规则影响。

## 余额兜底

余额兜底是用户级开关，默认关闭。字段设计：

- `users.subscription_balance_fallback_enabled`：是否开启“订阅消耗完时，自动消耗余额”。
- `users.subscription_balance_fallback_group_id`：用户或管理员选择的余额兜底目标分组。
- `users.subscription_balance_fallback_limit_usd`：余额兜底累计上限，必须大于 0 才会生效。
- `users.subscription_balance_fallback_used_usd`：已经通过订阅兜底消耗的余额金额。

本轮把上限定义为累计上限，不做每日或每月自动重置。用户关闭开关时不清空已使用值；用户调高上限后，剩余额度等于 `limit - used`。管理员可以单独调整已使用值或重置为 0。

余额兜底只在以下情况生效：

1. API Key 绑定的是订阅分组。
2. 用户在该订阅分组下没有可继续消耗的同产品族产品额度，或者产品额度不足。
3. 用户已开启余额兜底。
4. `subscription_balance_fallback_limit_usd > subscription_balance_fallback_used_usd`，也就是剩余累计兜底预算大于 0。
5. `users.subscription_balance_fallback_group_id` 已设置。
6. 选中的兜底分组是活跃标准分组，也就是 `groups.subscription_type = 'standard'`。
7. 用户对该标准分组有权限，权限判断与普通标准 API Key 可见分组规则一致。
8. 请求开始前用户余额不是负数。

兜底请求进入余额计费模式：本次 usage log 的 `billing_type` 为余额模式，不再写入产品订阅用量，不再消耗旧版订阅限额。请求运行分组切换到用户选择的标准余额分组，实际扣费使用该标准分组倍率和用户专属分组倍率，避免订阅分组倍率影响余额价格。

由于实际费用只有请求结束后才能准确计算，系统在请求前只能做“是否还有兜底预算、目标分组是否合法、用户余额是否非负”的资格检查。扣费落库时必须原子扣减余额并累加 `subscription_balance_fallback_used_usd`，同时校验不会超过累计上限。若一次请求的实际费用超过剩余额度，扣费事务应失败并返回计费错误。

余额计费允许当前已结算请求把 `users.balance` 扣成负数，但负数余额会阻止后续请求，直到用户充值。

### 兜底分组来源

新的闭环行为使用 `users.subscription_balance_fallback_group_id` 作为唯一兜底目标来源。用户或管理员必须显式选择一个有权限的活跃标准分组。

历史字段 `groups.balance_fallback_group_id` 可以保留为兼容配置数据，但新用户可控兜底不应使用它做决策，也不能用订阅分组上的映射替用户选择余额分组。

本轮可以保留或创建余额模式 `team/plus` 混合号池，倍率为 1 倍，作为用户可选的标准余额分组。迁移不应批量更改用户现有 API Key 或账号池。

## 前端体验

用户订阅页需要同时完成销售说明和运行状态说明：

- 展示产品订阅支持多分组、共享额度和昨日结转。
- 每个产品卡片展示总每日进度条，包含昨日结转。
- 进度条下方展示今日额度、昨日结转、昨日结转剩余和各分组倍率。
- 明确标注每日窗口按北京时间 UTC+8 00:00 重置，周/月窗口是滚动 7/30 天。
- 激活产品或兑换产品成功后提示用户去生成新的产品订阅分组 API Key。
- 增加默认关闭的开关“订阅消耗完时，自动消耗余额”。
- 开关开启时必须要求用户选择一个有权限的标准余额分组，并设置正数累计上限。
- 展示兜底已使用、剩余预算和余额为负时的阻断提示。

API Key 创建或编辑页需要：

- 选择标准分组时不展示产品族选择。
- 选择订阅分组且只有一个可用产品族时，可以自动绑定该产品族。
- 选择订阅分组且有多个可用产品族时，必须要求用户选择产品族，并写入 `api_keys.subscription_product_family`。

管理员侧需要能继续：

- 创建产品订阅码时绑定明确 `product_id`。
- 查看所有产品订阅用户。
- 查看用户产品订阅自激活以来的用量统计，包括昨日结转组成。
- 在产品订阅列表执行调整、额度重置和撤销。
- 管理用户余额兜底：开关、累计上限、已使用值、重置已使用值和兜底分组。

管理员调整产品订阅时不提供用户级每日限额覆盖；额度应来自产品定义和订阅实例窗口状态。

## 迁移兼容

- 旧 API Key 仍按真实 `group_id` 鉴权。绑定订阅分组的旧 Key 可以命中新版产品订阅上下文；如果同一分组存在多个产品族，旧 Key 因缺少 `subscription_product_family` 会被要求重新选择或编辑。
- 未使用的产品订阅兑换码如果缺少 `product_id`，兑换必须失败为配置错误，不再通过 `group_id` 或绑定关系猜测产品。
- 未使用的旧版订阅兑换码继续按旧版订阅处理，不受产品 `product_id` 强制规则影响。
- 已迁移到新版产品订阅的用户，前端优先展示新版产品订阅，旧版订阅只作为兼容信息保留。
- 不在本轮迁移账号池和 API Key，不批量更改用户现有 Key 的 `group_id`。
- 新增 `users.subscription_balance_fallback_group_id` 和 `api_keys.subscription_product_family` 应为可空字段；既有用户默认不启用兜底，已经开启兜底的用户也必须选择兜底分组后才会实际生效。
- 保留 `groups.balance_fallback_group_id`，但它不再作为新用户可控兜底的决策来源。

## 测试计划

后端测试：

- 产品订阅兑换码缺少 `product_id` 时兑换失败，即使存在 `group_id`。
- 管理员生成产品订阅码缺少 `product_id` 时失败。
- 产品绑定拒绝标准分组，只允许活跃订阅分组。
- 订阅分组解析失败不回退到旧版订阅。
- 用户同一分组下有多个产品族且 API Key 未绑定产品族时返回歧义错误。
- API Key 绑定产品族后只在该族内选择产品。
- 用户同一分组同产品族下有两个活跃产品，先选择排序/激活更早的产品；该产品额度耗尽后自动选择下一个产品。
- 用户同一分组下有不同产品族产品，不跨产品族自动切换。
- 产品选择不再按最晚过期时间决定。
- 余额兜底默认关闭时，产品额度耗尽仍返回限额错误。
- 余额兜底开启、有正数剩余累计预算、选择有权限活跃标准分组时，订阅分组请求切换到用户选择的余额分组并按余额计费。
- 余额兜底选择未授权分组、非活跃分组或订阅分组时失败。
- 请求开始前用户余额为负数时，兜底请求被阻止。
- 余额兜底扣费事务同时扣余额和累加 `subscription_balance_fallback_used_usd`，超过累计上限时失败。
- 已结算的余额请求可以把余额扣成负数，后续请求被阻止。
- 北京时间 00:00 日窗口重置和昨日结转正确。
- 周窗口按订阅实例窗口开始滚动 7 天，月窗口按订阅实例窗口开始滚动 30 天。
- 管理员调整、重置额度和撤销产品订阅接口可用。
- 管理员可以设置兜底开关、累计上限、已使用值和兜底分组。

前端测试：

- 订阅页展示包含结转的每日进度条和组成说明。
- 订阅页标注北京时间日窗口和滚动 7/30 天周/月窗口。
- 订阅页展示余额兜底开关，默认关闭。
- 开启兜底后必须选择标准授权分组并设置正数累计上限，保存后重新加载仍保持。
- 兑换产品成功后提示生成新的产品订阅分组 API Key。
- 管理员产品订阅码生成必须选择产品。
- API Key 创建/编辑页只在订阅分组需要产品族时展示并要求选择。
- 管理员产品订阅列表的调整、重置、撤销按钮调用已实现后端接口。
- 管理员用户或订阅详情页可管理余额兜底设置。
